# Unity SRPG项目架构设计思路

## 🏗️ 第一步：确定整体架构方向

### 设计思路：为什么选择MVC + 模块化？

**背景分析**：
- 这是一个回合制策略游戏，涉及战斗、UI、关卡、配置等多个子系统
- 需要支持多人协作开发
- 项目需要长期维护和扩展

**架构选择理由**：
1. **MVC模式**：天然适合游戏开发，数据、界面、逻辑分离
2. **模块化设计**：功能独立，便于团队协作和后期维护
3. **事件驱动**：降低模块间耦合度
4. **分层架构**：便于性能分析和优化

**架构层次设计**：
```
┌─────────────────────────────────────────┐
│              游戏表现层                  │
│  (UI界面、动画、特效、音效等)           │
├─────────────────────────────────────────┤
│              业务逻辑层                  │
│  (战斗逻辑、关卡逻辑、游戏规则等)       │
├─────────────────────────────────────────┤
│              核心系统层                  │
│  (管理器、工具类、通用系统等)           │
├─────────────────────────────────────────┤
│              数据层                     │
│  (配置文件、游戏数据、资源数据等)       │
└─────────────────────────────────────────┘
```

## 🔧 第二步：搭建核心框架基础

### 1. 创建单例管理器基类

**设计思路**：游戏需要全局访问的管理器（如战斗管理、UI管理等），但直接创建多个单例会导致管理混乱。

**解决方案**：泛型单例基类 + GameApp统一入口

```csharp
// 泛型单例基类，提供类型安全的单例实现
public class Singleton<T> where T : new() {
    private static T _instance;
    private static readonly object _lock = new object();

    public static T Instance {
        get {
            if (_instance == null) {
                lock (_lock) {
                    if (_instance == null) {
                        _instance = new T();
                    }
                }
            }
            return _instance;
        }
    }
}

// GameApp统一入口，避免分散的单例访问
public class GameApp {
    public static FightManager FightManager => Singleton<FightManager>.Instance;
    public static ViewManager ViewManager => Singleton<ViewManager>.Instance;
    public static EventCenter EventCenter => Singleton<EventCenter>.Instance;
    public static ConfigManager ConfigManager => Singleton<ConfigManager>.Instance;
    // ... 其他管理器
}
```

**为什么这样设计**：
- ✅ 统一的访问入口，避免`SomeManager.Instance`分散在各处
- ✅ 线程安全的单例实现
- ✅ 便于后续替换为依赖注入

## 🔄 第三步：建立模块通信机制

### 设计思路：模块间如何通信？

**问题分析**：
- 战斗系统需要通知UI系统更新界面
- UI系统需要触发战斗系统的操作
- 关卡系统需要监听战斗结果
- 直接调用会导致高度耦合

**解决方案**：三层事件系统

```csharp
// 事件中心，统一管理事件通信
public class EventCenter {
    // 普通事件：持久化监听，适合系统级事件
    private Dictionary<string, UnityAction<object>> eventDict;

    // 临时事件：一次性监听，适合异步回调
    private Dictionary<string, UnityAction<object>> tempEventDict;

    // 对象事件：跟随对象生命周期，适合UI组件
    private Dictionary<object, Dictionary<string, UnityAction<object>>> objEventDict;

    // 普通事件注册和触发
    public void AddEvent(string eventName, UnityAction<object> callback) {
        if (eventDict.ContainsKey(eventName)) {
            eventDict[eventName] += callback;
        } else {
            eventDict.Add(eventName, callback);
        }
    }

    public void BroadcastEvent(string eventName, object arg = null) {
        if (eventDict.ContainsKey(eventName)) {
            eventDict[eventName]?.Invoke(arg);
        }
    }

    // 临时事件：触发后自动移除
    public void AddTempEvent(string eventName, UnityAction<object> callback) {
        tempEventDict[eventName] = callback;
    }

    public void BroadcastTempEvent(string eventName, object arg = null) {
        if (tempEventDict.ContainsKey(eventName)) {
            tempEventDict[eventName]?.Invoke(arg);
            tempEventDict.Remove(eventName); // 触发后移除
        }
    }
}

// 使用示例：战斗系统触发事件
public class FightController : BaseController {
    public override void Init() {
        // 注册事件监听
        GameApp.EventCenter.AddEvent("Fight.Start", OnFightStart);
        GameApp.EventCenter.AddEvent("Hero.Died", OnHeroDied);
    }

    private void OnHeroDied(object data) {
        var hero = data as Hero;
        // 通过事件通知UI更新，无需直接调用UI组件
        GameApp.EventCenter.BroadcastEvent("UI.UpdateHeroList", hero);
    }
}
```

**为什么这样设计**：
- ✅ 松耦合：模块间通过事件通信，无需知道对方存在
- ✅ 灵活性：支持一对多、多对一通信
- ✅ 生命周期管理：三种事件类型适应不同场景
- ✅ 性能优化：事件缓存和批量处理

### 1. 战斗系统设计

**设计思路**：回合制战斗需要清晰的状态流转和实体管理

```csharp
// 战斗状态机，管理战斗流程
public class FightManager {
    public GameState state = GameState.Idle;
    private FightUnitBase curr; // 当前战斗单元

    public void ChangeState(GameState newState) {
        state = newState;

        // 根据状态创建对应的战斗单元
        switch (state) {
            case GameState.Idle:
                curr = new FightIdle();
                break;
            case GameState.Enter:
                curr = new FightEnter();
                break;
            case GameState.Player:
                curr = new FightPlayerUnit();
                break;
            case GameState.Enemy:
                curr = new FightEnemyUnit();
                break;
        }

        curr.Init();
    }

    public void Update(float dt) {
        if (curr != null) {
            if (curr.Update(dt)) {
                // 当前状态完成，根据逻辑切换到下一状态
                HandleStateComplete();
            }
        }
    }
}

// 战斗实体基类
public abstract class ModelBase {
    public int rowIndex, colIndex;  // 地图位置
    public string id;               // 唯一标识
    public int hp, maxHp;          // 生命值
    public bool IsStop { get; set; }

    // 距离计算，用于攻击范围和移动判定
    public float GetDist(ModelBase other) {
        int dx = Mathf.Abs(rowIndex - other.rowIndex);
        int dy = Mathf.Abs(colIndex - other.colIndex);
        return Mathf.Sqrt(dx * dx + dy * dy);
    }
}
```

**为什么这样设计**：
- ✅ 状态清晰：每个战斗阶段都有独立的处理逻辑
- ✅ 易于扩展：新增状态只需继承FightUnitBase
- ✅ 时间控制：支持定时状态切换和异步操作
- ✅ 实体管理：统一的战斗实体基类，便于扩展

### 2. UI管理系统设计

**设计思路**：40+个UI界面需要统一的生命周期管理和层级控制

```csharp
// 视图管理器，统一管理所有UI界面
public class ViewManager {
    private Dictionary<int, IBaseView> _opens;     // 当前打开的视图
    private Dictionary<int, IBaseView> _viewCache;  // 视图缓存
    private Dictionary<int, ViewInfo> _views;       // 视图配置信息

    // 视图信息配置
    public class ViewInfo {
        public string prefabName;     // 预制体名称
        public Transform parentTf;    // 父级变换
        public BaseController controller; // 关联的控制器
        public int sortingOrder;      // 显示层级
    }

    // 打开视图的统一接口
    public void Open(ViewType viewType, params object[] args) {
        int key = (int)viewType;

        // 检查是否已打开
        if (_opens.ContainsKey(key)) {
            var view = _opens[key];
            view.SetVisible(true);
            view.Open(args);
            return;
        }

        // 检查缓存
        if (_viewCache.ContainsKey(key)) {
            var view = _viewCache[key];
            _opens.Add(key, view);
            view.SetVisible(true);
            view.Open(args);
            return;
        }

        // 创建新视图
        CreateView(viewType, args);
    }

    private void CreateView(ViewType viewType, params object[] args) {
        var viewInfo = _views[(int)viewType];

        // 实例化预制体
        GameObject instance = GameObject.Instantiate(
            Resources.Load<GameObject>($"View/{viewInfo.prefabName}"),
            viewInfo.parentTf
        );

        // 设置层级
        var canvas = instance.GetComponent<Canvas>();
        if (canvas != null) {
            canvas.sortingOrder = viewInfo.sortingOrder;
        }

        // 初始化视图
        var baseView = instance.GetComponent<IBaseView>();
        baseView.ViewID = (int)viewType;
        baseView.Controller = viewInfo.controller;
        baseView.InitUI();
        baseView.InitData();
        baseView.Open(args);

        _opens.Add((int)viewType, baseView);
    }
}

// 视图类型枚举，统一管理所有界面
public enum ViewType {
    // UI界面
    BeginView = 1,
    SettingsView = 2,
    MessageView = 3,

    // 战斗界面
    FightView = 10,
    FightSelCharView = 11,
    HeroDesView = 12,
    EnemyDesView = 13,
    SelectOptionView = 14,

    // 关卡界面
    SelectLevelView = 20,

    // 加载界面
    LoadView = 30,

    // 战斗操作界面
    DragHeroView = 40,
}
```

**为什么这样设计**：
- ✅ 统一管理：所有UI界面通过ViewManager统一管理
- ✅ 生命周期清晰：创建、显示、隐藏、销毁流程明确
- ✅ 层级控制：通过sortingOrder控制显示层级
- ✅ 缓存机制：避免重复创建，提升性能
- ✅ 配置化：视图信息与代码分离，便于维护

// 配置管理器，统一管理所有游戏配置
public class ConfigManager {

### 1. 事件系统性能优化

**设计思路**：事件系统使用频繁，需要避免GC和性能瓶颈

// 事件参数对象池，减少GC
public class EventDataPool {
    private static Queue<EventData> _pool = new Queue<EventData>();

    public struct EventData {
        public int IntValue;
        public float FloatValue;
        public string StringValue;
        public Vector3 Position;
    }

    public static EventData Get() {
        if (_pool.Count > 0) {
            return _pool.Dequeue();
        }
        return new EventData();
    }

    public static void Release(EventData data) {
        // 重置数据
        data.IntValue = 0;
        data.FloatValue = 0f;
        data.StringValue = null;
        _pool.Enqueue(data);
    }
}

// 事件调试器，监控事件性能
public class EventDebugger {
    private static List<EventLog> _eventLogs = new List<EventLog>();

    public class EventLog {
        public string EventName;
        public DateTime Timestamp;
        public int ListenerCount;
    }

    public static void LogEvent(string eventName, int listenerCount) {
        var log = new EventLog {
            EventName = eventName,
            Timestamp = DateTime.Now,
            ListenerCount = listenerCount
        };

        _eventLogs.Add(log);

        // 保持日志数量在合理范围内
        if (_eventLogs.Count > 1000) {
            _eventLogs.RemoveAt(0);
        }
    }
}

### 2. 寻路算法优化

**设计思路**：AI寻路是性能热点，需要优化算法效率

```csharp
// A*寻路算法优化实现
public class AStarPathfinding {
    private List<Block> FindPath(Block start, Block target) {
        var openSet = new List<Block>();
        var closedSet = new HashSet<Block>();
        var cameFrom = new Dictionary<Block, Block>();

        // 使用曼哈顿距离作为启发函数（比欧几里得距离计算更快）
        var gScore = new Dictionary<Block, float>();
        var fScore = new Dictionary<Block, float>();

        openSet.Add(start);
        gScore[start] = 0;
        fScore[start] = GetManhattanDistance(start, target);

        while (openSet.Count > 0) {
            // 优化：使用最小堆或排序列表
            Block current = GetLowestFScore(openSet, fScore);

            if (current == target) {
                return ReconstructPath(cameFrom, current);
            }

            openSet.Remove(current);
            closedSet.Add(current);

            foreach (var neighbor in GetNeighbors(current)) {
                if (closedSet.Contains(neighbor)) continue;

                float tentativeGScore = gScore[current] + 1; // 假设移动成本为1

                if (!openSet.Contains(neighbor)) {
                    openSet.Add(neighbor);
                } else if (tentativeGScore >= gScore[neighbor]) {
                    continue;
                }

                cameFrom[neighbor] = current;
                gScore[neighbor] = tentativeGScore;
                fScore[neighbor] = gScore[neighbor] + GetManhattanDistance(neighbor, target);
            }
        }

        return null; // 未找到路径
    }

    private float GetManhattanDistance(Block a, Block b) {
        return Mathf.Abs(a.rowIndex - b.rowIndex) + Mathf.Abs(a.colIndex - b.colIndex);
    }
}
```
